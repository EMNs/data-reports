Airflow / dbt Modernisation Report

Date: 2026-03-12  ·  Repos: terraform-aws-live-airflow, dbt  ·  Reference: Vinted Engineering — Orchestrating Success

Executive Summary

Our current Airflow/dbt setup runs dbt as coarse selector-level tasks (p3, p5) inside a monolithic dbt_analytics_transform DAG. This means a single failing model can block unrelated downstream work, cross-team dependencies are manually wired and often implicit, and SLA metadata defined in dpr.yml files has no operational effect.

We are already mid-migration to a data product model — 58 dpr.yml files exist across the dbt repo with ownership, SLA, and asset metadata. This migration is not parallel to the modernisation work described here; it is the foundation of it. The dpr.yml spec, combined with a DAG generator, is the path to a Vinted-standard setup.

Target State

One Airflow DAG per data product, generated from dpr.yml + dbt manifest.json. Teams own a data contract (dpr.yml), not a Python DAG file. The platform handles scheduling, sensor wiring, alert routing, and SLA enforcement automatically.

Requirements

1 Task-Per-Model DAG Generator

Build a generator that reads each dpr.yml and the dbt manifest.json to produce one Airflow DAG per data product, with one task per model and lineage-preserving task dependencies.

  • Replace the monolithic dbt_analytics_transform p3/p5 selector tasks
  • A failing model in the board product does not affect the user product
  • Teams add models by updating assets.dbt in their dpr.yml — no airflow repo PR needed
  • Platform changes (Airflow version upgrades) require only generator updates, not per-DAG changes
2 Data Product Spec Completeness & Migration Strategy

A significant portion of the dbt repo — marts/, derived/, export/, shared/, sources/ — is not yet part of any data product. Blocking the generator on full migration completion would delay everything. Instead, adopt a two-tier model that lets the generator run for claimed and unclaimed models from day one.

Tier 1 — Proper data products (have a dpr.yml)

Full spec required: runbook, slack_channel_id, sla_expected_delivery_time_utc, sla_monthly_max_error_budget_in_hours, data_product_priority, owners, schedule. CI enforces all fields. Generator produces an isolated DAG per product.

Tier 2 — Unclaimed models (no dpr.yml yet)

A single catch-all _unclaimed.dpr.yml acts as a temporary home. The generator auto-populates its assets.dbt with any model not claimed by a proper product — teams do not maintain it manually. It has relaxed requirements: only schedule and slack_channel_id (pointing to a shared platform channel) are required. No SLA fields enforced.

# _unclaimed.dpr.yml  (maintained by platform team)
id: _unclaimed
display_name: Unclaimed Models (migration in progress)
assets:
  dbt: []  # auto-populated by generator from manifest
owners:
  - id: urn:li:corpGroup:data-platform
    type: TECHNICAL_OWNER
properties:
  schedule: "0 6 * * *"
  slack_channel_id: C_PLATFORM_CHANNEL
  # no SLA fields required — these are not yet products

How the two tiers coexist:

  • The generator runs product DAGs and the _unclaimed DAG independently — the monolith (dbt_analytics_transform) can be retired as soon as the generator is validated, not after full migration
  • CI warns (not errors) when a model is unclaimed — makes the remaining footprint visible without blocking teams
  • CI publishes a migration progress metric: number of models in _unclaimed shrinks over time as teams graduate models to proper products
  • When a team creates a dpr.yml and adds models to it, those models are automatically removed from _unclaimed on next generator run — no manual coordination needed
  • The _unclaimed DAG is deleted when it has zero assets

CI rules for proper products (Tier 1 only):

  • CI fails on PRs with missing required fields in any non-_unclaimed dpr.yml
  • Model-level ownership exists in meta config — dpr.yml becomes the authoritative source; CI validates that meta.owner matches the owning product
  • New models added to a Tier 1 product must satisfy the full spec — they cannot be added to _unclaimed if their domain already has a product
3 Schedule and Frequency Control

Teams must be able to declare how often their data product runs, and at what time, directly in dpr.yml. Today sla_delivery_cadence: daily and sla_expected_delivery_time_utc: 9 exist but are documentation — neither drives anything in Airflow.

properties:
  sla_delivery_cadence: daily
  sla_expected_delivery_time_utc: 9
  schedule: "0 6 * * *"   # ← new: drives the actual Airflow schedule
  • schedule accepts a cron string or shorthand (@daily, @hourly)
  • sla_expected_delivery_time_utc remains the SLA deadline — separate from when the DAG starts
  • For sub-daily products, sla_delivery_cadence supports hourly, every_6h, etc.
  • CI validates that schedule is consistent with sla_delivery_cadence
  • Changing run frequency is a one-line dpr.yml change — no airflow repo PR needed
4 SLA-Driven Airflow Configuration

Wire the remaining SLA fields already present in dpr.yml into the generated DAGs so they have operational effect.

  • sla_expected_delivery_time_utc → DAG deadline (distinct from schedule start time)
  • sla_monthly_max_error_budget_in_hours → SLA miss tracking
  • slack_channel_id → failure and SLA miss alert routing (replaces hardcoded webhook URLs)
  • data_product_priority → sensor timeouts and alert priority
5 Automated Cross-Product ExternalTaskSensors

When a model in product A refs a model owned by product B, the generator automatically inserts an ExternalTaskSensor — no manual wiring.

  • Sensors operate at task level (specific model task), not DAG level
  • Engineers declare dependencies via dbt ref() as they do today; the platform translates these into sensors
  • Removes the need for manually maintained sensor blocks for dbt-to-dbt dependencies
6 External Source Dependency Graph (Airbyte and other integrations)

Cross-product sensors only cover dbt-to-dbt dependencies. Many data products also depend on raw data populated by external import DAGs — Airbyte syncs for Salesforce, Google Search Console, etc. Today these are manually wired with hardcoded DAG IDs and task IDs.

"This DAG is a dependency for dbt_analytics_transform. Changes to schedule_interval or task_ids here need to be reflected in the corresponding sensor." — comment in airbyte_salesforce_import.py

Solution: a Source Registry

# source_registry.yml
salesforce:
  dag_id: airbyte_salesforce_import
  task_id: End
  schedule: "0 3 * * *"

google_search_console:
  dag_id: airbyte_google_search_console_import
  task_id: End
  schedule: "0 2 * * *"
  • The generator reads source() calls from the manifest, looks them up in the registry, generates sensors automatically
  • When an import DAG changes its task ID or schedule, the Source Registry is updated once — all sensors pick up the change on next deploy
  • CI validates that every source() call has a corresponding registry entry
  • Source Registry is owned by the data-platform team, not individual product teams
7 Circular Dependency Detection in CI

Cross-product ref() cycles cannot be caught by dbt but will cause deadlocks in Airflow.

  • The generator builds the full cross-product dependency graph before emitting any DAG
  • A topological sort is run on that graph; if it fails, CI fails and outputs the cycle path
  • All cross-product sensors must be generated — manually authored sensors are prohibited so cycles remain detectable
8 CI Downstream Impact Analysis at the Product Level

Extend the existing Acryl impact analysis to surface which data products and teams are affected by a schema change, not just which models.

  • PR comments include: affected data product name, owning team, Slack channel
  • CI validates that cross-product sensor references point to tasks that actually exist in the target product's generated DAG
9 Standardise DAG Authoring
  • dpr.yml becomes the only entry point for scheduling dbt models
  • A linting rule detects create_dbt_task() calls outside the generator
  • A small set of approved patterns enforced via pre-commit for non-dbt DAGs
  • Runbook link enforced as a required field in dpr.yml

Benefits

⚡

Faster incident recovery

One failing model no longer blocks an entire priority tier. Failure blast radius shrinks from "half the warehouse" to one data product.

🔔

Alerts route to the right team immediately

slack_channel_id in dpr.yml routes failures directly to the owning team's channel. No triage, no "who owns this?" thread.

🛡️

Schema changes stop silently breaking downstream consumers

CI identifies affected data products and their owners before a PR merges. The conversation happens pre-merge, not post-incident.

📋

SLAs become operational commitments

sla_expected_delivery_time_utc today is documentation. Once wired to the generator, Airflow enforces deadlines against the monthly error budget.

🔄

Airflow upgrades stop being migration projects

With a generator, all DAGs are derived. An Airflow version upgrade is a generator change, not a 118-file PR.

🚀

New data products are cheap to ship

One dpr.yml file replaces a Python DAG, catalog-info.yml, sensor wiring, alert config, and a PR in the airflow repo.

🔗

Coordination debt is eliminated

Cross-product dependencies declared in dbt ref() automatically generate sensors. The dependency graph in Airflow always matches dbt.

High-Level Delivery Phases

1
Stabilise the Data Product Spec & Introduce the Catch-All
Prerequisite for everything else — does not require full migration to complete
  • Audit all 58 existing dpr.yml files for missing required fields (Tier 1)
  • Define the full required field set for Tier 1 products
  • Create _unclaimed.dpr.yml as the catch-all for all models in marts/, derived/, export/, shared/ and other unclaimed directories
  • Add CI enforcement for Tier 1 products (all required fields must be present)
  • Add CI warning (not error) for models landing in _unclaimed — makes migration debt visible
  • Add CI migration progress check: report count of claimed vs unclaimed models on each PR
  • Teams with models in _unclaimed can graduate them to a proper dpr.yml at any point — no central coordination required
Deliverable: All models are schedulable via the generator (either in a proper product or _unclaimed). Migration to proper products can continue incrementally after the generator goes live.
2
Build the DAG Generator
Core engineering work
  • Generator reads dpr.yml + manifest.json, produces one DAG per product with task-per-model
  • SLA fields → schedule, timeouts, sla_miss_callback
  • slack_channel_id → alert routing
  • Cross-product ref() → ExternalTaskSensor generation
  • Circular dependency detection — fail CI if a cross-product cycle is found
Deliverable: Generator produces valid, equivalent DAGs from any dpr.yml + manifest.
3
Shadow Mode Validation
De-risk before production
  • Deploy generated DAGs alongside existing ones with paused schedules
  • Compare task graphs between generated and hand-authored DAGs
  • Run generated DAGs manually against dev/staging
  • Identify and resolve any gaps
Deliverable: Confidence that generated DAGs are equivalent to or better than current production.
4
Incremental Migration
Phased production rollout
  • Migrate 2–3 low-risk data products first (simple lineage, few downstream consumers)
  • Monitor for one full week per wave — SLA compliance, failure routing, sensor behaviour
  • Roll out remaining products in priority order
  • Deprecate and delete dbt_analytics_transform
Deliverable: dbt_analytics_transform deleted; all dbt scheduling lives in generated product DAGs.
5
Upgrade CI Impact Analysis
Can run in parallel with Phase 4 later waves
  • Extend Acryl impact analysis to join impacted models to dpr.yml — surfaces product name, owner, Slack channel
  • Add cross-product sensor reference validation to CI
Deliverable: Schema change PRs automatically notify the right teams before merge.
6
Remove Manual DAG Authoring for dbt
Can run in parallel with Phase 4 later waves
  • Policy: no new hand-authored Python DAG files for dbt models
  • Linting rule detects create_dbt_task() calls outside the generator
  • Delete hand-authored dbt task code from the airflow repo
Deliverable: The airflow repo contains no dbt task logic; engineers only touch dpr.yml.
